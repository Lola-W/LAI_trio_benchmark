# Reference-Panel Definition and Masking Strategy

This document defines the preprocessing contracts for LAI benchmarking before
running FLARE and RFMix.

## Input Source

- Pedigree file: `1kGP.3202_samples.pedigree_info.txt`
- Complete-trio child definition: `fatherID != 0` and `motherID != 0`

## Generated Core Sets

All sets are generated by:

```bash
bash code/scripts/build_sample_sets.sh
```

Output directory: `code/meta/`

Core files:
- `trios.complete.tsv`: `trio_id	child_id	father_id	mother_id`
- `samples.children.tsv`: complete-trio children
- `samples.parents.tsv`: unique referenced parents
- `samples.trio_members.tsv`: unique union of children and their parents
- `samples.all_3202.tsv`: all pedigree sample IDs

## Benchmark Conditions

### Condition A: Strict Masking (default for Mendelian consistency)

Definition:
- Query set: all trio members (`query.strict.tsv`)
- Reference set: all non-query samples (`reference.strict.tsv`)

Reason:
- Prevents reference leakage from any member of the evaluated trios.
- Supports parent-child transmission checks because child and both parents are in query inference set.

### Condition B: Child-Only Masking (leakage sensitivity track)

Definition:
- Query set: complete-trio children only (`query.child_only.tsv`)
- Reference set: all non-child samples (`reference.child_only.tsv`)

Reason:
- Matches the original design where the child is excluded but parents may remain in reference.
- Useful as a stress-test for sensitivity to reference leakage.

## Labeling Contract for FLARE / RFMix

Population labels are not present in the pedigree file. Provide a separate metadata table:

- Required columns: `sample_id` and `label`
- Label can be:
  - Population (fine-grained), or
  - Superpopulation (AFR/EUR/EAS/SAS/AMR)

From that table, derive tool-specific maps in the next stage:
- FLARE: `<sample_id> <panel_name>`
- RFMix: `<sample_id>\t<label>`

The reference map must be filtered to the selected reference set (`reference.strict.tsv` or `reference.child_only.tsv`) for each benchmark condition.

## Runtime Best Practices (OOM-Safe Child-Only Runs)

These workflow defaults now prioritize stable completion on chr22 while
preserving the masking contract:

- `run_rfmix_test_child_only.sh` consumes:
  - `MAX_REF_PER_POP` (cap reference samples per label)
  - `REF_MIN_AF` (optional reference MAF thinning)
- Snakemake passes these from `code/snake_conf.yaml`:
  - `rfmix_max_ref_per_pop`
  - `rfmix_ref_min_af`

Current OOM-safe defaults in `snake_conf.yaml`:

- `threads.rfmix: 14`
- `rfmix_max_ref_per_pop: 40`
- `rfmix_ref_min_af: 0.01`

To restore full-panel benchmark behavior, set:

- `rfmix_max_ref_per_pop: 0`
- `rfmix_ref_min_af: 0`
